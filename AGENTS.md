# AGENTS.md

## 목표
이 프로젝트는 현재 목표 기능을 구현하는 데에는 충분하다.  
에이전트의 역할은 기능을 새로 덧붙이는 것보다, 기존 기능을 유지하면서 프로젝트의 완성도를 높이는 것이다.

특히 아래 3가지를 우선적으로 개선한다.

1. 효율성
2. 가독성
3. 유지보수성

## 작업 원칙
- 기존에 동작하는 핵심 기능은 훼손하지 않는다.
- 단순히 코드를 많이 바꾸는 것보다, 적은 변경으로 효과가 큰 개선을 우선한다.
- 동작은 같더라도 구조가 더 명확하고 이해하기 쉬운 방향을 선택한다.
- 임시방편식 수정 대신, 반복되는 문제를 줄일 수 있는 구조적 개선을 우선한다.
- 성능, 메모리 사용, 불필요한 연산, 중복 로직이 보이면 적극적으로 정리한다.
- 함수, 클래스, 모듈의 역할이 불명확하면 책임이 드러나도록 분리하거나 이름을 개선한다.
- 매직 넘버, 중복 조건문, 중복 데이터 가공, 과도한 중첩 로직은 가능한 한 제거한다.

## 우선순위
에이전트는 아래 순서로 판단하고 작업한다.

1. 기존 기능이 깨지지 않도록 안정성을 먼저 확인한다.
2. 비효율적인 로직, 불필요한 반복, 과도한 복잡도를 줄인다.
3. 코드 흐름과 책임 분리를 개선해 읽기 쉽게 만든다.
4. 필요한 경우에만 파일 구조, 모듈 경계, 인터페이스를 정리한다.
5. 변경 이유와 효과가 드러나도록 결과를 설명한다.

## 구체적인 개선 지침
- 동일하거나 유사한 로직이 여러 곳에 있으면 공통화 여부를 검토한다.
- 한 함수가 너무 많은 역할을 수행하면 더 작은 단위로 나눈다.
- 변수명, 함수명, 클래스명은 역할과 의도가 드러나게 수정한다.
- 조건문이 복잡하면 조기 반환, 보조 함수 분리, 명확한 분기 구조로 단순화한다.
- 데이터 흐름이 불분명하면 입력, 변환, 출력 단계를 분리한다.
- 성능에 영향을 주는 반복 계산, 불필요한 객체 생성, 과도한 탐색을 점검한다.
- 주석은 최소한으로 사용하되, 코드만으로 의도가 드러나지 않는 경우에만 짧고 명확하게 작성한다.

## 금지 사항
- 겉보기에만 깔끔해지는 불필요한 대규모 리팩터링은 피한다.
- 근거 없이 추상화를 늘리거나 계층을 추가하지 않는다.
- 기능 변경이 필요한 경우, 반드시 이유와 영향 범위를 먼저 명확히 한다.
- 프로젝트 전반의 스타일을 흔드는 일관성 없는 수정은 하지 않는다.

## 결과물 요구사항
에이전트는 작업 후 아래 내용을 분명히 정리한다.

- 무엇을 개선했는지
- 왜 그 변경이 필요한지
- 기능 유지에 어떤 점을 확인했는지
- 성능, 가독성, 유지보수성 측면에서 어떤 이점이 있는지
- 추가로 남아 있는 리스크나 후속 개선 포인트가 무엇인지

## 판단 기준
아래 질문에 많이 "예"라고 답할 수 있을수록 좋은 개선이다.

- 같은 기능을 더 단순한 구조로 설명할 수 있는가?
- 처음 보는 사람이 더 빠르게 이해할 수 있는가?
- 수정 시 부작용이 생길 가능성이 줄어드는가?
- 실행 비용이나 불필요한 처리량이 감소하는가?
- 이후 기능 확장 시 재사용하기 쉬운 구조인가?
